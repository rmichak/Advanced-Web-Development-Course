Express vs Next.js: Routing

In Express, you define routes in code: app.get('/tasks', ...). In Next.js, routes are determined by your folder structure. A file at app/api/tasks/route.js automatically handles requests to /api/tasks. No manual route registration needed.

Square Brackets Matter!

The [id] folder must have actual square brackets in the name. This tells Next.js it's a dynamic segment — a part of the URL that can be any value, like a task ID. It's the same concept as :id in Express routes, just expressed through the folder name instead of code. In the terminal, escape the brackets with backslashes: \[id\].

Create Helper Functions

Build the same read/write helpers, but with ES module syntax.

Open lib/tasks.js and add the file storage utilities:

// lib/tasks.js
import fs from 'fs/promises';
import path from 'path';

const TMP_FILE = '/tmp/tasks.json';
const SEED_FILE = path.join(process.cwd(), 'tasks.json');

export async function readTasks() {
  try {
    const data = await fs.readFile(TMP_FILE, 'utf8');
    return JSON.parse(data);
  } catch {
    try {
      const seed = await fs.readFile(SEED_FILE, 'utf8');
      return JSON.parse(seed);
    } catch {
      return [];
    }
  }
}

export async function writeTasks(tasks) {
  await fs.writeFile(TMP_FILE, JSON.stringify(tasks, null, 2));
}

ES Modules vs CommonJS

Notice import/export instead of require/module.exports. Next.js uses ES modules (the newer JavaScript module system) by default. The logic is identical to your Express helpers — just different syntax for loading and sharing code between files.

process.cwd() vs __dirname

Express used __dirname (the directory of the current file). Next.js uses process.cwd(), which returns the current working directory — basically the root folder of your project. Both point to where tasks.json lives, but Next.js's module system doesn't support __dirname the same way.

Why /tmp Instead of Writing Directly to tasks.json?

Vercel's deployed environment is read-only — your project files (including tasks.json) can't be modified at runtime. If you try to write to tasks.json on Vercel, you'll get a 500 Internal Server Error. Writing to /tmp/ works because Vercel allows temporary file storage there. The code reads seed data from your project's tasks.json as a fallback, but writes all changes to /tmp/tasks.json. Note that /tmp data is ephemeral — it won't persist between serverless function invocations, so data may reset. In a real production app, you'd use a database. This works fine for our assignment.

Create Collection Routes (GET all & POST)

Handle requests to /api/tasks — retrieving all tasks and creating new ones.

Open app/api/tasks/route.js and add the GET and POST handlers:

// app/api/tasks/route.js
import { NextResponse } from 'next/server';
import { readTasks, writeTasks } from '@/lib/tasks';

// GET /api/tasks — Return all tasks
export async function GET() {
  const tasks = await readTasks();
  return NextResponse.json(tasks);
}

// POST /api/tasks — Create a new task
export async function POST(request) {
  const { text } = await request.json();

  if (!text) {
    return NextResponse.json(
      { error: 'Text is required' },
      { status: 400 }
    );
  }

  const newTask = {
    id: Date.now(),
    text,
    completed: false,
    createdAt: new Date().toISOString()
  };

  const tasks = await readTasks();
  tasks.push(newTask);
  await writeTasks(tasks);

  return NextResponse.json(newTask, { status: 201 });
}

Key Differences from Express

Function names ARE the methods: Export a function called GET and it handles GET requests. Export POST for POST requests. No app.get() or app.post() needed — Next.js routes by function name.

NextResponse instead of res: Instead of res.json(data), you return NextResponse.json(data). Instead of res.status(201).json(data), pass status as a second argument: NextResponse.json(data, { status: 201 }).

The @ Import Alias

@/lib/tasks is a shortcut (called an import alias) that Next.js sets up for you automatically. The @ symbol represents your project root, so @/lib/tasks means "the tasks.js file in the lib folder at the project root." Without it, you'd have to write relative paths like ../../../lib/tasks, which get messy fast.

Create Single Item Routes (GET one, PUT, DELETE)

Handle requests to /api/tasks/[id] — operations on a specific task.

Open app/api/tasks/[id]/route.js and add all three handlers:

// app/api/tasks/[id]/route.js
import { NextResponse } from 'next/server';
import { readTasks, writeTasks } from '@/lib/tasks';

// GET /api/tasks/[id] — Return a single task
export async function GET(request, { params }) {
  const { id } = await params;
  const parsedId = parseInt(id);
  const tasks = await readTasks();
  const task = tasks.find(t => t.id === parsedId);

  if (!task) {
    return NextResponse.json({ error: 'Not found' }, { status: 404 });
  }

  return NextResponse.json(task);
}

// PUT /api/tasks/[id] — Update a task
export async function PUT(request, { params }) {
  const { id } = await params;
  const parsedId = parseInt(id);
  const updates = await request.json();
  const tasks = await readTasks();
  const index = tasks.findIndex(t => t.id === parsedId);

  if (index === -1) {
    return NextResponse.json({ error: 'Not found' }, { status: 404 });
  }

  tasks[index] = { ...tasks[index], ...updates, id: parsedId };
  await writeTasks(tasks);
  return NextResponse.json(tasks[index]);
}

// DELETE /api/tasks/[id] — Remove a task
export async function DELETE(request, { params }) {
  const { id } = await params;
  const parsedId = parseInt(id);
  const tasks = await readTasks();
  const filtered = tasks.filter(t => t.id !== parsedId);

  if (filtered.length === tasks.length) {
    return NextResponse.json({ error: 'Not found' }, { status: 404 });
  }

  await writeTasks(filtered);
  return new Response(null, { status: 204 });
}

Important: await params

In Next.js 15+, params is a Promise and must be awaited: const { id } = await params. If you skip the await, you'll get an error. This is different from older tutorials you might find online.

Express vs Next.js: Reading URL Parameters

In Express: req.params.id. In Next.js: const { id } = await params (from the second function argument). Same data, different access pattern.

204 No Content in Next.js

NextResponse.json() always sends a body, so for 204 (no body) we use the standard Web API: new Response(null, { status: 204 }). Next.js runs on web standards under the hood.

About File Storage on Vercel

Vercel runs your API as serverless functions — instead of a single server running 24/7 like your local Express server, each request spins up a temporary instance of your code that shuts down after it responds. The /tmp storage we use is ephemeral — data may not persist between requests if Vercel spins up a new instance. That's expected! For this assignment, just test all endpoints quickly in a row. In a real app, you'd use a database instead of a JSON file.

Express vs Next.js — What You Learned

Now that you've built the same API twice, here's how the two approaches compare.

Feature Express Next.js
Routing app.get('/tasks', ...) File: app/api/tasks/route.js
HTTP method app.get(), app.post() export function GET(), POST()
URL params req.params.id const { id } = await params
Request body req.body await request.json()
JSON response res.json(data) NextResponse.json(data)
Status codes res.status(201).json() NextResponse.json(data, {status: 201})
Module system CommonJS (require) ES Modules (import)
Deployment Any Node.js host Vercel (one-click), serverless

Which Should You Use?

Express is great for standalone APIs where you need full control over your server, middleware, and deployment. Next.js is ideal when your API is part of a larger web app with both frontend pages and backend endpoints. Both are widely used — knowing both makes you more versatile.
