What You'll Learn

This is your first backend project. You'll build an API (Application Programming Interface) — a program that listens for HTTP requests and sends back JSON data. This is how every web and mobile app talks to its server. The patterns you learn here work the same way in other Node.js frameworks like NestJS, Hono, and Fastify.

Create Your Project

Set up a new Node.js project folder and install dependencies.

Open your terminal and create a new project folder. Run npm init -y to create a package.json with default settings:

$ mkdir express-tasks-api
$ cd express-tasks-api

Wrote to /your/path/express-tasks-api/package.json

Now install your dependencies. Express is the web framework that handles routing and requests. cors (Cross-Origin Resource Sharing) lets other websites or apps make requests to your API — without it, browsers will block those requests for security reasons. nodemon auto-restarts the server when you save changes:

$ npm install express cors
$ npm install nodemon --save-dev

Open package.json and update the "scripts" section so you can start the server with npm run dev:

"scripts": {
  "dev": "nodemon index.js",
  "start": "node index.js"
}

Why nodemon?

Without nodemon, you'd have to stop and restart the server every time you change your code. Nodemon watches your files and restarts automatically — it's a huge time saver during development.

Set Up the Server

Create index.js with Express, middleware, and helper functions for reading/writing data.

Middleware is code that runs between receiving a request and sending a response — think of it as a processing step that every request passes through before reaching your routes. Create index.js and add the imports, middleware, and two helper functions that read and write to tasks.json:

// index.js
const express = require('express');
const cors = require('cors');
const fs = require('fs').promises;
const path = require('path');

const app = express();
const PORT = 3000;
const DATA_FILE = path.join(__dirname, 'tasks.json');

// Middleware — these run on EVERY request before your routes
app.use(cors());            // Allow requests from other origins
app.use(express.json());   // Parse JSON request bodies

// Helper: Read all tasks from the JSON file
async function readTasks() {
  try {
    const data = await fs.readFile(DATA_FILE, 'utf8');
    return JSON.parse(data);
  } catch {
    return [];  // If file doesn't exist, return empty array
  }
}

// Helper: Write tasks array to the JSON file
async function writeTasks(tasks) {
  await fs.writeFile(DATA_FILE, JSON.stringify(tasks, null, 2));
}

CommonJS vs ES Modules

Express projects use require() (CommonJS syntax). This is different from the import syntax you'll see in Next.js. Both do the same thing — load modules — just with different syntax. Express defaults to CommonJS because it predates ES modules in Node.js.

Middleware Order Matters

app.use(express.json()) must come before your routes. Without it, req.body will be undefined when you try to read JSON from POST and PUT requests.

4

Add GET Routes

Create two GET endpoints — one to retrieve all tasks, one to retrieve a single task by ID.

Add these routes after your helper functions in index.js:

// GET /tasks — Return all tasks
app.get('/tasks', async (req, res) => {
  try {
    const tasks = await readTasks();
    res.json(tasks);
  } catch (error) {
    res.status(500).json({ error: 'Failed to read tasks' });
  }
});

// GET /tasks/:id — Return a single task by ID
app.get('/tasks/:id', async (req, res) => {
  try {
    const id = parseInt(req.params.id);
    const tasks = await readTasks();
    const task = tasks.find(t => t.id === id);

    if (!task) {
      return res.status(404).json({ error: 'Task not found' });
    }

    res.json(task);
  } catch (error) {
    res.status(500).json({ error: 'Failed to read task' });
  }
});

URL Parameters

The :id in /tasks/:id is a URL parameter. Express captures whatever value appears in that position and puts it in req.params.id. So a request to /tasks/123 gives you req.params.id = "123". Note it's always a string, which is why we use parseInt() to convert it to a number.

Always Handle 404

If a client asks for a task that doesn't exist, return 404 Not Found — not 200 OK with null. Proper status codes make your API predictable and easier to debug.

5

Add POST Route

Create an endpoint that accepts JSON data and creates a new task.

// POST /tasks — Create a new task
app.post('/tasks', async (req, res) => {
  try {
    const { text } = req.body;

    // Validate: text is required
    if (!text) {
      return res.status(400).json({ error: 'Text is required' });
    }

    const newTask = {
      id: Date.now(),                         // Unique ID from timestamp
      text,
      completed: false,
      createdAt: new Date().toISOString()     // ISO format timestamp
    };

    const tasks = await readTasks();
    tasks.push(newTask);
    await writeTasks(tasks);

    res.status(201).json(newTask);            // 201 = Created
  } catch (error) {
    res.status(500).json({ error: 'Failed to create task' });
  }
});

Why 201 Instead of 200?

201 Created tells the client that a new resource was successfully created. It's more specific than a generic 200 OK. We also return the created task (with its new id) so the client knows exactly what was stored.

Input Validation

Always validate incoming data. If someone sends a request without text, we return 400 Bad Request instead of creating an empty task. This prevents bad data from getting into your storage.

6

Add PUT and DELETE Routes

Add the last two CRUD operations — updating existing tasks and removing them.

// PUT /tasks/:id — Update an existing task
app.put('/tasks/:id', async (req, res) => {
  try {
    const id = parseInt(req.params.id);
    const tasks = await readTasks();
    const index = tasks.findIndex(t => t.id === id);

    if (index === -1) {
      return res.status(404).json({ error: 'Task not found' });
    }

    // Merge existing task with updates, preserve original id
    tasks[index] = { ...tasks[index], ...req.body, id };
    await writeTasks(tasks);

    res.json(tasks[index]);
  } catch (error) {
    res.status(500).json({ error: 'Failed to update task' });
  }
});

// DELETE /tasks/:id — Remove a task
app.delete('/tasks/:id', async (req, res) => {
  try {
    const id = parseInt(req.params.id);
    const tasks = await readTasks();
    const filtered = tasks.filter(t => t.id !== id);

    if (filtered.length === tasks.length) {
      return res.status(404).json({ error: 'Task not found' });
    }

    await writeTasks(filtered);
    res.sendStatus(204);   // 204 = No Content (success, no body)
  } catch (error) {
    res.status(500).json({ error: 'Failed to delete task' });
  }
});

The Spread Operator for Updates

{ ...tasks[index], ...req.body, id } merges the existing task with whatever fields the client sends. The id at the end ensures the ID can't be overwritten. This lets clients send partial updates — just {"completed": true} without resending the entire task.

204 No Content

DELETE returns 204 with no response body. This is standard REST practice — the task is gone, there's nothing to return. The absence of a body confirms the deletion succeeded.

Set Up Your Collection

  1. Open Postman (download from postman.com if you haven't)
  2. Click Collections in the left sidebar → click +
  3. Name it: Express Tasks API

Test Each Endpoint

Work through these in order. Save each request to your collection after testing it.

1. Create a Task

Set method to POST, URL to http://localhost:3000/tasks. Go to Body tab → select raw → choose JSON from the dropdown:

{
  "text": "Learn REST APIs"
}

Click Send. Expected: 201 with the created task (including its id). Copy the id value — you'll need it for the next tests. Create 2-3 more tasks so you have data to work with.

2. Get All Tasks

GET http://localhost:3000/tasks — no body needed. Expected: 200 with an array of all your tasks.

3. Get Single Task

GET http://localhost:3000/tasks/YOUR_TASK_ID — replace YOUR_TASK_ID with an actual id from step 1. Expected: 200 with that one task. Also try a fake ID like 9999999 to verify you get 404.

4. Update a Task

PUT http://localhost:3000/tasks/YOUR_TASK_ID with this body:

{
  "completed": true
}

Expected: 200 with the updated task showing "completed": true.

5. Delete a Task

DELETE http://localhost:3000/tasks/YOUR_TASK_ID — no body. Expected: 204 with no response body. Run GET all tasks again to confirm it's gone.

Save Each Request!

After testing each endpoint, click Save (Ctrl+S) and add it to your "Express Tasks API" collection. You need to export this collection for submission.
