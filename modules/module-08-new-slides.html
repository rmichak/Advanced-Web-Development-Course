How OAuth 2.0 Works

What happens when a user clicks "Sign in with Google" — the Authorization Code Grant flow.

1

Your App → Google

Redirect user to Google's auth server with your client_id and requested scopes (e.g., email, profile)

2

User Logs In on Google's Page

User enters credentials on Google's site (not yours!) and approves permissions. Your app never sees their password.

3

Google → Your App (Authorization Code)

Google redirects back to your redirect_uri with a short-lived authorization code in the URL

4

Your Server → Google (Token Exchange)

Server-to-server request: exchange the code + client_secret for access & refresh tokens. Secret never goes to browser.

5

Access User Info via API

Use the access token to call Google's API and get user profile (name, email, avatar). User is now authenticated.

OpenID Connect (OIDC)

An identity layer on top of OAuth 2.0. OAuth = "what can I access?" OIDC adds "who am I?" — standardizes user identity with an ID token.

SSL/TLS & HTTPS

How data is encrypted in transit between the browser and server.

🔐 What is TLS?

Transport Layer Security — encrypts all data between browser and server. SSL is the old name (deprecated 2015). We use TLS 1.2 and 1.3 today.

📜 Certificates & CAs

Issued by Certificate Authorities (Let's Encrypt, DigiCert). Proves the server is who it claims to be. That padlock icon in your browser? That's a valid TLS certificate.

🛡️ HSTS

HTTP Strict Transport Security — a header that tells browsers "always use HTTPS for this domain." Prevents downgrade attacks.

The TLS Handshake (Simplified)

🌐 Browser connects to server
↓
📜 Server sends its TLS certificate
↓
✅ Browser verifies certificate with CA
↓
🤝 They agree on encryption method
↓
🔒 Encrypted tunnel established

⚠️ Without HTTPS

Passwords, tokens, and API keys travel in plain text. Anyone on the same WiFi with Wireshark can read everything.

✅ Good News

Vercel, Netlify, and Cloudflare Pages give you HTTPS automatically via Let's Encrypt. Free, automated, zero config.

Encryption Fundamentals

The four building blocks every developer needs to understand.

Type How It Works Reversible? Examples Used For
🔑 Symmetric Encryption Same key encrypts and decrypts Yes (with key) AES-256 Data at rest, TLS tunnel (after handshake)
🔐 Asymmetric Encryption Public key encrypts, private key decrypts Yes (with private key) RSA, ECC TLS handshake, JWT signing, key exchange
#️⃣ Hashing One-way function → fixed-length output No — irreversible SHA-256, bcrypt Passwords, data integrity checks
📝 Encoding Format conversion — no key involved Yes (by anyone) Base64, URL encoding JWT payloads, data transport format

⚠️ Encoding ≠ Encryption

Base64 (used in JWT payloads) is not secure. Anyone can decode it — no key needed. Never put secrets in a JWT thinking they're hidden.

⚠️ Hashing ≠ Encryption

Encryption is reversible (with the key). Hashing is not. That's why passwords are hashed — even if the database is stolen, passwords can't be recovered.

How They Work Together in TLS

Asymmetric encryption secures the handshake (key exchange) → then symmetric encryption (AES) handles the fast data transfer. Best of both worlds.

How Passwords Are Actually Stored

Best practices for credential storage — the #1 thing to never get wrong.

🚫 NEVER Store Plaintext Passwords

Not in a database. Not in a text file. Not anywhere. If your database is breached and passwords are in plaintext, every account is instantly compromised. Period.

#️⃣ Hashing

Run password through bcrypt or Argon2 → store the hash. On login, hash the input and compare hashes. One-way — can't be reversed.

🧂 Salting

Add a random string (salt) to each password before hashing. Same password → different hashes. Defeats rainbow table attacks (pre-computed hash lookups).

⚡ What Supabase Does

Uses bcrypt with automatic salting. You call signUp() and it handles everything. You never touch the raw password.

Storing Secrets & Tokens

What Where to Store Why
API Keys & Secrets Environment variables (.env.local) Never in source code or git — use .gitignore
Production Secrets Secrets manager (AWS Secrets Manager, HashiCorp Vault) Encrypted, audited, access-controlled
Tokens (Browser) localStorage (simple), httpOnly cookies (better), in-memory (most secure) Supabase uses localStorage by default — reasonable trade-off

💡 Personal Security Tip

Use a credential manager like 1Password or Bitwarden for your own passwords. Never reuse passwords across sites. This isn't just advice for users — it's critical for developers managing dozens of API keys and service accounts.
