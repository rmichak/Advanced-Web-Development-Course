HTTP Protocol

The Language of the Web

HTTP Protocol Deep Dive

Understanding the Foundation

Key Characteristics

  • Stateless Protocol - Each request is independent
  • Text-Based Format - Human-readable for debugging
  • Request-Response Model - Client initiates, server responds
  • Extensible - Custom headers enable new features

Protocol Evolution

HTTP/1.0New TCP per request
HTTP/1.1Keep-alive, pipelining
HTTP/2Binary, multiplexed
HTTP/3QUIC, UDP-based

HTTP/2 Benefits: Header compression (HPACK), server push, stream prioritization, single connection multiplexing

HTTP Methods

CRUD Operations and RESTful Design

GET READ

Retrieve resource. Safe, idempotent, cacheable.

POST CREATE

Submit data. Not safe, not idempotent.

PUT UPDATE

Replace entire resource. Idempotent.

DELETE DELETE

Remove resource. Idempotent.

PATCH

Partial update to resource

HEAD

GET without body (metadata only)

OPTIONS

Discover allowed methods (CORS)

Method Properties

Safe Methods

GET, HEAD, OPTIONS - No side effects on server

Idempotent Methods

GET, PUT, DELETE, HEAD - Same result on repeat

Cacheable Methods

GET, HEAD - Response can be stored

HTTP Status Codes

Success and Redirection

2xx Success SUCCESS

200OK - Standard success
201Created - Resource created
204No Content - Success, empty body
206Partial Content - Range request

3xx Redirection REDIRECT

301Moved Permanently
302Found (Temporary)
304Not Modified (use cache)
307Temporary Redirect

Caching Tip: 304 Not Modified saves bandwidth by telling clients to use cached versions

HTTP Status Codes

Client and Server Errors

4xx Client Errors CLIENT

400Bad Request
401Unauthorized
403Forbidden
404Not Found
429Too Many Requests

5xx Server Errors SERVER

500Internal Server Error
502Bad Gateway
503Service Unavailable
504Gateway Timeout

Debugging Tip: 4xx = check your request (URL, headers, body). 5xx = check server logs.

Response Headers

Security Essentials

Headers and Vulnerabilities

Security Headers

CORS Controls cross-origin API access
CSP Prevents XSS attacks
HSTS Forces HTTPS connections
Same-Origin Browser sandbox for scripts

Common Vulnerabilities

  • XSS - Cross-Site Scripting
  • CSRF - Cross-Site Request Forgery
  • SQL Injection - Database attacks
  • Clickjacking - UI deception

Best Practice: Use security headers, validate all input, escape output, keep dependencies updated.

Key Takeaways

What to Remember

1. HTTP is the Foundation

Understanding request/response cycles, methods, and status codes is essential for debugging and building web applications.

2. Web Servers Do More Than Serve

Beyond serving files, they handle security, caching, load balancing, and act as the gateway to your application.

3. Choose Stacks Wisely

Technology choices should align with team skills and project needs rather than trends.

4. Security is Non-Negotiable

HTTPS everywhere, security headers, and understanding vulnerabilities should be part of every developer's toolkit.

5. Deploy Early, Deploy Often

GitHub Pages makes it easy to share your work. Build the habit of publishing your projects.
